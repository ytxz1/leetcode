# 1833. 雪糕的最大数量

题目链接：[LeetCode 1833. 雪糕的最大数量](https://leetcode.cn/problems/maximum-ice-cream-bars/)

## 题目描述

夏天到了，小男孩准备购买雪糕。

数组：

```text
costs
```

表示每支雪糕的价格，其中：

```text
costs[i]
```

表示第 `i` 支雪糕需要花费的硬币数量。

小男孩一共有：

```text
coins
```

枚硬币。

每支雪糕最多只能购买一次，请计算在总花费不超过 `coins` 的情况下，最多可以买多少支雪糕。

## 题目真正要优化的目标

本题要求：

```text
购买雪糕的数量尽可能多
```

它不是要求：

```text
让硬币刚好花完
```

也不是要求：

```text
买到的雪糕总价格最大
```

例如：

```text
costs = [1, 2, 5]
coins = 5
```

如果买价格为 `5` 的雪糕：

```text
花费 = 5
数量 = 1
```

如果买价格为 `1` 和 `2` 的雪糕：

```text
花费 = 3
数量 = 2
剩余硬币 = 2
```

虽然第二种方案没有把硬币花完，但买到了更多雪糕，所以第二种方案更好。

这说明：

```text
不能只追求花费最大，要追求购买数量最大。
```

## 核心观察

每支雪糕对答案的贡献都相同：

```text
买到一支，数量就增加 1
```

不同之处只有价格。

因此，为了在固定预算中容纳尽可能多的雪糕，应该优先购买价格最低的雪糕。

核心策略：

```text
1. 把价格从小到大排序
2. 从最便宜的开始买
3. 买得起就扣钱并增加数量
4. 买不起当前雪糕时直接停止
```

这是一种排序贪心。

## 图解：为什么先买便宜的

假设：

```text
costs = [4, 1, 3, 1, 2]
coins = 7
```

排序前：

```text
雪糕价格：[4] [1] [3] [1] [2]
```

排序后：

```text
雪糕价格：[1] [1] [2] [3] [4]
            ↑   ↑   ↑   ↑
            先购买便宜的雪糕
```

用 `■` 表示一枚硬币。

初始预算：

```text
coins = 7

预算：■■■■■■■
```

购买第一支，价格 `1`：

```text
花费：■
剩余：■■■■■■
数量：1
```

购买第二支，价格 `1`：

```text
花费：■
剩余：■■■■■
数量：2
```

购买第三支，价格 `2`：

```text
花费：■■
剩余：■■■
数量：3
```

购买第四支，价格 `3`：

```text
花费：■■■
剩余：空
数量：4
```

价格为 `4` 的雪糕买不起。

最终买到：

```text
4 支
```

## 为什么贪心正确

### 直观解释

购买任何一支雪糕，答案都只增加 `1`。

既然收益相同，就应该优先选择花费更小的雪糕。

花费越小：

- 越节省预算
- 越有机会继续购买后面的雪糕
- 不会减少最终购买数量

### 最便宜的 k 支具有最小总价

把所有价格排序：

```text
c1 <= c2 <= c3 <= ... <= cn
```

如果想购买恰好 `k` 支雪糕，那么价格最小的 `k` 支：

```text
c1 + c2 + ... + ck
```

一定是所有“购买 `k` 支”方案中总价最小的。

因此：

```text
如果最便宜的 k 支都超过预算，
那么其他任意 k 支也一定超过预算。
```

反过来：

```text
如果最便宜的 k 支没有超过预算，
就一定可以买到至少 k 支。
```

所以，只需要从小到大依次购买，直到预算不足。

### 交换论证

假设某个最优方案购买了一支价格较高的雪糕 `expensive`，却没有购买一支价格更低的雪糕 `cheap`：

```text
cheap <= expensive
```

把 `expensive` 换成 `cheap`：

```text
购买数量不变
总花费不会增加
```

交换后仍然是可行方案，并且可能剩下更多硬币。

不断进行这种交换，最终一定可以得到一个“优先选择便宜雪糕”的最优方案。

这就是贪心选择的正确性依据。

## 代码变量说明

核心代码：

```cpp
sort(costs.begin(), costs.end());

int count = 0;

for (int cost : costs) {
    if (coins < cost) {
        break;
    }

    coins -= cost;
    count++;
}

return count;
```

变量含义：

| 变量 | 含义 |
|---|---|
| `costs` | 所有雪糕的价格，排序后为升序 |
| `coins` | 当前剩余硬币数量 |
| `cost` | 当前正在考虑购买的雪糕价格 |
| `count` | 已经成功购买的雪糕数量 |

## 为什么买不起当前雪糕就可以停止

代码：

```cpp
if (coins < cost) {
    break;
}
```

因为 `costs` 已经升序排列。

如果当前价格是：

```text
cost = 5
```

当前剩余：

```text
coins = 3
```

那么当前雪糕买不起。

后面的价格满足：

```text
后续价格 >= 5
```

剩余硬币只有 `3`，所以后面的雪糕也全部买不起。

因此没有必要继续遍历，可以直接 `break`。

## 带详细注释的代码

完整代码见：

[solution.cpp](D:\leetcode\2026.6\1833. 雪糕的最大数量\solution.cpp)

LeetCode 提交时只需要：

```cpp
class Solution {
public:
    int maxIceCream(vector<int>& costs, int coins) {
        sort(costs.begin(), costs.end());

        int count = 0;

        for (int cost : costs) {
            if (coins < cost) {
                break;
            }

            coins -= cost;
            count++;
        }

        return count;
    }
};
```

## 正确性证明

将价格排序为：

```text
c1 <= c2 <= ... <= cn
```

算法从 `c1` 开始依次购买。

假设算法最终买了前 `k` 支：

```text
c1 + c2 + ... + ck <= coins
```

但下一支买不起：

```text
c1 + c2 + ... + ck + c(k+1) > coins
```

价格最小的 `k + 1` 支雪糕的总价已经超过预算。

任何其他 `k + 1` 支雪糕的总价，都不会小于最便宜的 `k + 1` 支总价。

所以不存在能够购买 `k + 1` 支的方案。

算法已经买到 `k` 支，因此 `k` 就是最大数量。

## 循环不变量

每轮循环开始前，可以认为：

```text
1. costs 中当前 cost 前面的雪糕已经处理完。
2. count 等于已经购买的雪糕数量。
3. coins 等于购买这些雪糕后剩余的硬币。
4. 已购买的是目前处理范围中最便宜的雪糕。
```

如果可以买当前 `cost`：

```text
coins -= cost
count++
```

上述关系继续成立。

如果买不起：

```text
coins < cost
```

由于后面的价格都不低于 `cost`，后面也全部买不起，算法可以正确结束。

## 复杂度分析

设雪糕数量为 `n`。

排序：

```text
O(n log n)
```

遍历：

```text
O(n)
```

总时间复杂度：

```text
O(n log n)
```

空间复杂度：

```text
O(log n)
```

主要来自 C++ `sort` 的递归栈。

代码会原地修改 `costs` 的顺序，没有新建长度为 `n` 的辅助数组。

## 另一种方法：计数排序

题目中的雪糕价格有明确上限时，还可以统计每种价格出现多少次。

假设：

```text
frequency[p]
```

表示价格为 `p` 的雪糕数量。

然后从价格 `1` 开始购买。

对于价格 `price`：

```text
最多能买 coins / price 支
实际存在 frequency[price] 支
```

所以本价格能购买：

```text
buy = min(frequency[price], coins / price)
```

更新：

```text
coins -= buy * price
count += buy
```

代码示例：

```cpp
class Solution {
public:
    int maxIceCream(vector<int>& costs, int coins) {
        int maxCost = *max_element(costs.begin(), costs.end());
        vector<int> frequency(maxCost + 1, 0);

        for (int cost : costs) {
            frequency[cost]++;
        }

        int count = 0;

        for (int price = 1; price <= maxCost; price++) {
            int buy = min(frequency[price], coins / price);

            coins -= buy * price;
            count += buy;

            if (coins < price) {
                // 这里不能总是直接结束，因为下一轮价格更高，
                // 此时确实也不可能再购买，可以 break。
                break;
            }
        }

        return count;
    }
};
```

设最大价格为 `C`。

计数排序方案复杂度：

```text
时间复杂度：O(n + C)
空间复杂度：O(C)
```

当 `C` 不大时，这种方法可以比比较排序更快。

## 两种方法对比

| 方法 | 时间复杂度 | 空间复杂度 | 优点 |
|---|---:|---:|---|
| 排序贪心 | `O(n log n)` | `O(log n)` | 简单、通用、容易理解 |
| 计数排序 | `O(n+C)` | `O(C)` | 价格范围较小时更快 |

对于学习和面试，优先掌握排序贪心。

## 常见错误

### 错误 1：按照原顺序购买

例如：

```text
costs = [5,1,1,1]
coins = 5
```

如果按原顺序先买 `5`：

```text
只能买 1 支
```

排序后先买三个 `1`：

```text
可以买 3 支
```

所以必须先排序。

### 错误 2：优先购买最贵的

本题目标是数量最大。

购买昂贵雪糕会更快消耗预算，通常会减少数量。

### 错误 3：判断写成 `coins <= cost`

错误：

```cpp
if (coins <= cost) {
    break;
}
```

当：

```text
coins == cost
```

当前雪糕明明可以买，错误写法却会停止。

正确：

```cpp
if (coins < cost) {
    break;
}
```

### 错误 4：扣钱后忘记增加数量

购买成功后需要同时：

```cpp
coins -= cost;
count++;
```

### 错误 5：买不起当前雪糕后继续无意义遍历

因为数组已经升序，当前买不起，后面也买不起。

可以直接：

```cpp
break;
```

### 错误 6：把剩余硬币当作答案

题目返回的是：

```text
购买数量 count
```

不是剩余硬币 `coins`。

### 错误 7：误以为必须花完所有硬币

只要购买数量最大即可，剩余硬币可以大于 `0`。

## 通过这道题可以学习到什么

### 1. 学会识别贪心问题

本题有这些特征：

```text
每个物品收益相同：都贡献 1 支
每个物品成本不同：价格不同
预算有限
目标是选择尽可能多的物品
```

这种情况下，自然应该优先选择成本最低的物品。

### 2. 学会先明确优化目标

算法选择取决于目标。

如果目标是：

```text
数量最多
```

就优先买便宜的。

如果目标变成：

```text
总价值最大
```

就不一定能用相同策略，可能需要背包动态规划。

所以解题前要先问：

```text
题目到底要最大化什么？
```

### 3. 学会排序后简化决策

排序前，价格顺序混乱，每一步都不知道哪个最便宜。

排序后：

```text
最优选择总是在数组前面
```

于是复杂的选择过程变成一次线性扫描。

这种技巧在很多题中出现：

- 分发饼干
- 救生艇
- 活动选择
- 区间调度
- 最小成本选择

### 4. 学会交换论证

贪心算法不能只凭感觉。

本题可以通过交换说明：

```text
用便宜雪糕替换昂贵雪糕，
数量不变，花费不增加。
```

这是证明贪心正确性的经典方法。

### 5. 学会使用提前结束

排序后，一旦当前价格超过剩余预算：

```text
后面的价格只会更高
```

所以可以立即停止。

这是一种利用有序性的剪枝。

### 6. 学会维护剩余资源

代码中的：

```cpp
coins -= cost;
```

是在维护剩余资源。

类似变量还可能是：

- 剩余时间
- 剩余容量
- 剩余燃料
- 剩余操作次数

### 7. 学会用变量表调试

本题可以用四列检查每一轮：

```text
cost
操作前 coins
操作后 coins
count
```

一旦答案不对，很容易定位是：

- 排序没做
- 判断条件错
- 扣钱错
- 数量没增加

## 还可以延伸学习什么知识

### 1. 贪心算法

贪心的核心是：

```text
每一步选择当前看起来最优的方案，
并证明这些局部选择能组成全局最优解。
```

本题的局部最优选择：

```text
购买当前最便宜的雪糕
```

### 2. 计数排序

如果数据值范围不大，可以不用比较排序，而是统计每个价格出现次数。

这能把：

```text
O(n log n)
```

优化到：

```text
O(n + C)
```

### 3. 前缀和与二分查找

排序后还可以构造前缀和：

```text
prefix[k] = 最便宜的前 k 支总价
```

然后查找最大的 `k`，满足：

```text
prefix[k] <= coins
```

可以用二分查找。

不过本题只查询一次，直接遍历更简单。

如果有很多不同预算查询，前缀和加二分会很有价值。

### 4. 与 0-1 背包的区别

本题每支雪糕的收益都相同，都是数量 `1`，所以买最便宜的即可。

如果每支雪糕还有不同的快乐值：

```text
cost[i]  = 价格
value[i] = 快乐值
```

目标是最大快乐值，那么简单排序价格不一定正确，可能需要：

```text
0-1 背包动态规划
```

### 5. 与分数背包的区别

分数背包允许只取物品的一部分，通常按价值密度贪心。

本题每支雪糕只能：

```text
买一整支
或者不买
```

但因为每支贡献都相同，所以仍然可以按价格贪心。

### 6. 多查询问题

如果题目给出很多个 `coins` 查询，可以：

```text
排序 costs
计算前缀和
对每个 coins 二分查找
```

单次查询复杂度可以降为：

```text
O(log n)
```

### 7. 原地排序的副作用

代码：

```cpp
sort(costs.begin(), costs.end());
```

会修改输入数组。

LeetCode 本题允许这样做。

如果工程中还需要原顺序，可以先复制：

```cpp
vector<int> sortedCosts = costs;
sort(sortedCosts.begin(), sortedCosts.end());
```

## 学完后的自查问题

1. 为什么本题应该优先购买便宜的雪糕？
2. 为什么买不起当前雪糕后可以直接停止？
3. `count` 和 `coins` 分别表示什么？
4. 当 `coins == cost` 时能不能购买？
5. 为什么排序贪心的时间复杂度是 `O(n log n)`？
6. 如果所有雪糕收益不同，还能直接按价格排序吗？
7. 计数排序适合在什么情况下使用？
8. 如果有很多预算查询，可以怎样优化？

参考答案：

```text
1. 每支雪糕对数量的贡献相同，价格越低越节省预算。
2. 数组已经升序，后续价格不会低于当前价格。
3. count 是已购买数量，coins 是当前剩余硬币。
4. 可以，只有 coins < cost 时才买不起。
5. 排序需要 O(n log n)，遍历需要 O(n)。
6. 不一定，可能需要背包动态规划。
7. 当价格是整数且最大价格范围不大时。
8. 排序后计算前缀和，对每次预算使用二分查找。
```

## 总结

这道题可以记成四句话：

```text
目标是数量最大
价格升序排序
买得起就购买
买不起就停止
```

核心代码：

```cpp
sort(costs.begin(), costs.end());

for (int cost : costs) {
    if (coins < cost) {
        break;
    }

    coins -= cost;
    count++;
}
```

复杂度：

```text
时间复杂度：O(n log n)
空间复杂度：O(log n)
```

```text
排序贪心

交换论证

剩余资源维护
计数排序优化
前缀和与多查询扩展
```
